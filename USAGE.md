# PM Copilot 使用指南

给 Phalcon 产品经理的快速上手指南。

---

## 这是什么？

PM Copilot 是一个**内部知识系统**，帮助你：
1. 快速查阅**过去做过的产品决策**（比如"SAR 为什么是这样设计的"）
2. 一次性生成 PRD（包括自动插入相关截图和参考）
3. 减少重复解释，加快新 PM 上手

---

## 核心组件

### 1. **决策日志** (`decisions/`)
每次我们讨论 PRD 时，Claude Code 会自动记录：
- 你做了什么决策
- 为什么这样决策
- 假设是什么
- 有什么开放问题

**对你的意义：** 新 PM 不用每次都问你，直接看决策日志就能理解背景。

### 2. **产品事实库** (`facts/`)
- 各产品的定义、边界、支持的功能
- 共享术语表（SAR、Entity、Risk Score 等定义）
- 核心工作流说明

**对你的意义：** 每个新 PRD 前，我会读这些内容，问的问题更精准。

### 3. **截图/UI 索引** (`assets-index.md`)
当你提供产品截图或页面描述时，我会自动记录：
```
这是 Address Screening 的列表页（2026-04-15）
功能：批量选择、导出、分页
```

然后生成 PRD 时，我会自动在相关章节插入这些截图。

**对你的意义：** 你不用每次都截图、找截图，直接在对话中提一下。

## 怎么用？

### 场景 1：写一个新 PRD

```
你：「我想写一个地址筛选的批量导出功能」

我（Claude Code）会：
1. 读 facts/ 里关于 Address Screening 的定义
2. 读 decisions/ 里过去关于地址筛选的决策
3. 读 assets-index.md 里的截图
4. 问你关键问题（需求、边界、优先级）
5. 自动生成 PRD（带截图）
6. 生成一份决策日志存档

你：「加个批量操作的 undo 功能」
```

### 场景 2：新 PM 想了解某个模块

```
新 PM：「Address Screening 为什么是严格匹配，不支持模糊匹配？」

他可以：
- 读 facts/phalcon-compliance.md（产品定义）
- 读 decisions/ 里关于 Address Screening 的日志（找到原因）
- 看 assets-index.md 里的截图（理解当前 UI）

不用每次都打扰你。
```

### 场景 3：更新现有决策

```
你：「Address Screening 现在支持模糊匹配了，改了设计」
  [给我截图]

我会：
1. 更新 facts/
2. 更新 assets-index.md
3. 记录在新的决策日志里
```

### 场景 4：定期扫描和更新资产库

```
你：「/scan-assets」

我会：
1. 扫描 approved-prds/ 和 competitors/ 中的所有图片和 UI 描述
2. 自动更新 assets-index.md
3. 提取新的功能对标、UI 模式、竞品动态
4. 生成一份扫描报告，列出新增资产

这样你不用手动维护截图索引。
```

## 你需要做什么？

### 必做
1. 在 Claude Code 里和我讨论 PRD（这是现有流程，无改变）
2. **新增：** 当有新截图或 UI 改动时，在对话中提一下

### 可选
- 定期（比如月底）用 `/scan-assets` 命令自动扫描和更新资产库
- 定期检查 `decisions/` 里的日志，确认没有错漏
- 更新 `facts/` 里的产品定义（如果产品大改了）

## 常见问题

**Q: 我每次都要写详细描述吗？**  
A: 不用。简单说就行，比如 "这是地址列表页，新增了导出功能"。我会自动理解上下文。

**Q: 决策日志会很冗长吗？**  
A: 不会。一个决策日志通常 200-400 字，包括：决策+原因+假设+开放问题。短而精。

**Q: 这会影响现有工作流吗？**  
A: 不影响。你继续在 Claude Code 里写 PRD，只是多了一个资料库在后台积累。

**Q: 如果某个决策后来改了怎么办？**  
A: 新的决策日志会记录改动理由。历史决策仍然保留（方便追溯）。
