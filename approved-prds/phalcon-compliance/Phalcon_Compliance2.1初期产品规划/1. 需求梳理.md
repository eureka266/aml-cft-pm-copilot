## 1.2 核心用户角色

### 合规官

领导合规业务的开展，主导合规策略制定与监管沟通。非技术背景较多。

**痛点**

*   风险遗漏：未识别的高风险交易导致监管问询。
    
*   报告审核压力。
    

### 合规专员

负责合规业务的进展，处理日常的可疑报告提交等工作。非技术背景较多。

**痛点**

*   重复性操作没有便捷工具辅助的低效率性，如：Audit Log 导出、风险交易的 Note 记录。
    

### 风控团队

识别并管理市场、操作风险，保障平台财务安全。负责具体的用户封控、交易限制等操作。

在有的公司中风控人员和合规人员的职责有交叉部分。

# 产品定位

为Web3企业提供一站式风险监控与合规管理平台，覆盖安全攻击防御与交易合规需求。

目标市场：以香港、新加坡为重点。

## 关于合规

本产品“合规”指的都是符合 AML / CTF （反洗钱/反恐怖融资）相关规定，而实际上要完全满足 AML/CTF 合规所要采取的行动不止 KYT（还有 KYC 等）。

关于具体合什么规，可见：[《监管规定汇总》](https://alidocs.dingtalk.com/api/doc/transit?dentryUuid=ZX6GRezwJlqnOj3oHyxn0kZ98dqbropQ&queryString=utm_medium%3Ddingdoc_doc_plugin_card%26utm_source%3Ddingdoc_doc)。

# KYT 能力确定

## 如何融入 Phalcon 平台

### 差别与目标

| 维度 | Phalcon | KYT | 目标 |
| --- | --- | --- | --- |
| 用户角色 | 安全团队 | 合规/风控团队 | 界面隔离与数据协同的平衡：大型企业严格分离，中小企业融合操作。 |
| 用户频率 | 低频 | 高频 | 界面设计需区分核心工作流，避免高频用户被低频功能干扰。 |
| 核心行为 | 自动化 Action 以减少资金损失 | 风险处置、合规报告 | 在交互设计上要根据核心行为明确主要引导动作。 |
| 规则范围 | 每个 Monitor 单独设置有自己的规则 | 偏向于全局规则（覆盖所有地址/交易） | 设计上应不让用户产生混淆。 |

### Overlap

*   用户：即便使用 Phalcon 现有功能和 KYT 功能的主要用户群体不同，但是在工作流程上也会有需要协作的地方。
    
*   数据：部分地址、标签数据也会有需要共享的情况。
    

如果结合现在 Phalcon 平台功能考虑，对于现在每个模块需要做的改动如下：

| 模块 | 改动 |
| --- | --- |
| Pricing | *   CEX等客户需监控海量地址，按地址数计费不适用。<br>    <br>*   KYT 用户也不太会创建很多个 Monitor，按 Monitor 数量计费也不太适用。<br>    <br>需要单独设计 KYT Pricing Plan，参考竞品按照交易数量计费。 |
| Dashboard | *   展示的内容有区别，例如：合规用户不关注Mempool交易，需要增加 risk exposure 统计等。 |
| Alert Hub | *   Alert 列表的字段会有所区别（Detected Stage、User 等），不能共用一个列表，可能需要增加 tab 切换。<br>    <br>*   Alert Detail 抽屉中具体的展示内容要根据 Alert 是否由 KYT 产生进行区分。 |
| Addresses | *   导入的地址需要检测区分类型：protocol 还是 hot wallet。<br>    <br>*   另外如果是 hot wallet ，可能还需要展示关联的用户地址数量（同时提供其他导入方式选项）。 |
| Monitor | KYT 本身会提供默认模板，也会有 Custom Monitor。但通过竞品调研，觉得这种 Monitor 更应该是一种全局的 Rule ，制定后就会应用于所有导入的地址。 |
| Actions | KYT当前无 Action  需求。 |
| System | 若 Pricing 和用户权限部分有修改，则 System 下的 Organization 和 Usage 也需要相应修改。 |

除此之外，可能还要增加页面：

*   Customer/User： 展示交互/监控的用户列表。
    
*   Transactions：除了 Alert 列表，是否需要保留并展示所有扫描过的 transaction 信息（考虑到监管提出的“保留交易记录”要求）。（我们的风险评估也会有遗漏，但如果保留了交易记录，对于客户来说也还可以事后分析、形成报告）。
    

### 确定方案：区分 Workspace 

*   用户登录后需选择进入哪一个 Workspace，而 Workspace 分为 Security 和 Compliance 两种类型。
    
*   进入 Workspace 后，根据 Workspace 的类型展示相应的内容。
    
*   用户甚至可以设立多个 Compliance Workspace，以满足不同区域的政策需求：如某Workspace仅遵循 FATF，另一Workspace遵循新加坡 PSA、欧盟 MiCA。
    

#### 优缺点

✅ 优点：

*   交互设计会比较简单，单独考虑 KYT 需求，现在的 Phalcon 功能不需要有较大的更改。
    
    *   需要修改的地方：用户系统中需要增加 Workspace 创建和权限分配的部分。
        
    *   现在的 Phalcon 用户除需要建立 Workspace 外没有其他使用影响。

### Monitor 

所有的 Monitor 分为以下两种类型：

*   Transaction Monitor
    
*   Address Monitor

### Default Monitor

先只支持 FATF 的规定就可以。

包括：

*   FATF Rule - Exposure to illicit Entity  (Address / Incoming & Outgoing )
    
*   FATF Rule - VA Transactions to Online Gambling (Address / Incoming & Outgoing )
    
*   FATF Rule - VASPs with weak KYC (Transaction / Source of funds )
    
*   FATF Rule - Receipt of Funds from Mixing Services (Transaction / Source of funds )
